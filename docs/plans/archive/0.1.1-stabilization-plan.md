# PawnLogic 0.1.1 Stabilization Plan

Status: completed and archived. The `0.1.1` release is the current stable
0.1 line baseline; this document is historical release-planning context and
must not be used as an active implementation plan.

## Goal

Prepare a focused `0.1.1` patch release that fixes release metadata drift,
runtime configuration durability, shell export safety, and the highest-impact
silent failure paths found during the 0.1.0 audit.

The release should use multiple focused implementation branches if useful, but
only one final version bump and one public release tag.

## Current Audit Baseline

- Local full suite passes: `526 passed`.
- Ruff passes.
- Documentation heading structure check passes.
- Repository language policy check passes.
- GitHub raw `README.md` on `main` is updated for `0.1.0`.
- The public PyPI version badge resolves to `v0.1.0`.
- PyPI package metadata for `0.1.0` still contains the old README content and
  old `GUIDE_EN.md` documentation URL because the README and guide rename were
  merged after the `0.1.0` package was built and uploaded.

## Release Strategy

Use feature branches for reviewable fixes, merge them into `main`, then create
one final release branch for `0.1.1`.

Recommended branch chain:

1. `fix/security-docs-env-template`
2. `fix/runtime-atomic-writes`
3. `fix/provider-shell-key-quoting`
4. `fix/provider-session-error-visibility`
5. `refactor/provider-store-locking`
6. `release/0.1.1`

Only `release/0.1.1` should change the runtime version. Do not bump
`config/paths.py:VERSION` in the earlier fix branches.

## Task 2: Atomic Runtime File Writes

Priority: P1

Purpose:

Prevent truncated or partially written runtime config files when a process is
interrupted during writes.

Files:

- New helper location, preferably `core/file_store.py` or `config/io.py`
- `config/providers.py`
- `core/provider_runtime.py`
- `core/provider_tui.py`
- New or updated tests under `tests/`

Required changes:

- Add a small helper for atomic text writes:
  - Ensure the parent directory exists.
  - Write to a temporary file in the same directory.
  - Flush and fsync the temporary file.
  - Replace the destination with `os.replace`.
  - Preserve requested file permissions when needed.
- Use the helper for:
  - `custom_providers.json`
  - provider sync timestamps
  - provider model deletion from the TUI
  - `~/.pawnlogic/.env` key persistence
- Keep runtime paths under `PAWNLOGIC_HOME`.

Acceptance criteria:

- Existing provider behavior is unchanged.
- Invalid existing JSON is not silently overwritten without a warning.
- `.env` remains `0o600`.
- Tests prove writes use replace semantics and preserve key file permissions.

Focused checks:

```bash
python -m pytest tests/test_provider_runtime.py tests/test_providers.py -q --timeout=60
python -m pytest tests/test_provider_commands.py -q --timeout=60
python -m pytest tests/test_deployment_friendly.py -q --timeout=60
python -m ruff check .
git diff --check
```

## Task 3: Safe Shell Export Writing

Priority: P1

Purpose:

Prevent shell config corruption or command injection when the first-run wizard
writes API keys containing shell metacharacters.

Files:

- `core/commands/provider.py`
- Tests under `tests/test_provider_commands.py` or a narrower provider command
  test file

Required changes:

- Replace `export KEY="{raw}"` with a safe quoting strategy, preferably
  `shlex.quote`.
- Keep `os.environ[env_var] = key` unchanged for the current process.
- Avoid duplicate export lines for the same environment variable.
- Add tests for keys containing:
  - double quotes
  - dollar signs
  - backticks
  - single quotes
  - spaces

Acceptance criteria:

- The shell config line is syntactically safe.
- The in-process environment value remains the exact original key.
- Existing first-run wizard behavior remains unchanged for ordinary keys.

Focused checks:

```bash
python -m pytest tests/test_provider_commands.py -q --timeout=60
python -m pytest tests/test_deployment_friendly.py -q --timeout=60
python -m ruff check .
git diff --check
```

## Task 4: Provider And Session Error Visibility

Priority: P1

Purpose:

Make important user-facing failures visible without broadly rewriting every
defensive exception handler.

Files:

- `pawnlogic/cli.py`
- `config/providers.py`
- `core/provider_runtime.py`
- `core/provider_tui.py`
- `core/commands/provider.py`
- Tests under `tests/`

Required changes:

- Session resume:
  - If recent-session lookup or selected resume fails unexpectedly, log a
    warning.
  - In user-friendly mode, show a concise message and continue with a new
    session.
  - Do not crash startup.
- Provider JSON loading:
  - Keep malformed `custom_providers.json` from crashing startup.
  - Log a warning with the file path and parse error.
  - Avoid overwriting malformed JSON unless the user explicitly saves a new
    provider state.
- Provider TUI model deletion:
  - Replace silent `except Exception: pass` with a warning or visible status.
- Completer refresh after provider TUI:
  - Replace the broad `except Exception: pass` with a narrow `NameError`
    compatibility path plus warning for unexpected exceptions.

Acceptance criteria:

- Startup remains resilient.
- Important provider/session failures are visible in debug logs, and user mode
  receives concise actionable text where appropriate.
- Tests cover malformed JSON and resume failure paths.

Focused checks:

```bash
python -m pytest tests/test_provider_runtime.py tests/test_provider_commands.py -q --timeout=60
python -m pytest tests/test_deployment_friendly.py tests/test_session_utils.py -q --timeout=60
python -m ruff check .
git diff --check
```

## Task 5: Provider Store Locking And Direct Access Reduction

Priority: P2

Purpose:

Deepen the provider store interface so provider/model state changes have
locality and consistent snapshot semantics.

Files:

- `config/providers.py`
- `core/commands/provider.py`
- `core/provider_tui.py`
- `core/provider_runtime.py`
- Provider-related tests

Required changes:

- Add a re-entrant lock around provider/model mutations and snapshots.
- Keep `PROVIDERS` and `MODELS` as compatibility backing stores.
- Route new and touched code through:
  - `register_provider`
  - `remove_provider`
  - `register_model`
  - `remove_model`
  - `remove_models_for_provider`
  - `provider_snapshot`
  - `model_snapshot`
- Replace direct iteration over `PROVIDERS` / `MODELS` in provider command and
  TUI paths when the code can safely use snapshots.
- Keep the public behavior of `/provider`, `/model`, and completions unchanged.

Acceptance criteria:

- No runtime behavior drift in provider list, add, activate, fetch, model
  selection, or completions.
- Concurrent background TUI tasks read stable snapshots.
- Tests cover snapshot detachment and mutation helpers under the lock.

Focused checks:

```bash

python -m pytest tests/test_deployment_friendly.py -q --timeout=60
```

## Risks And Controls

- Do not try to rewrite PyPI `0.1.0` metadata. Publish `0.1.1` instead.
- Do not mix the version bump into early fix branches; it creates avoidable
  release conflicts.
- Do not broadly rewrite all `except Exception` handlers. Fix only
  user-visible paths with tests.
- Do not remove `PROVIDERS` or `MODELS`; keep them as compatibility backing
  stores until a later architecture release.
- Keep all non-`*_CN` files in English.
- Keep `CLAUDE.md` as a thin wrapper around `AGENT.md`.
