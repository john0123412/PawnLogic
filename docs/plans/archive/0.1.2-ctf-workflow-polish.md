# PawnLogic 0.1.2 CTF Workflow Polish Plan

Status: completed and archived. The 0.1.2 release has already shipped; this
document is historical release-planning context and must not be used as an
active implementation plan.

## Goal

PawnLogic 0.1.2 focuses on polishing the existing CTF workflow without changing
the product into an automated penetration-testing or security-assessment
platform.

The release should improve:

- CTF tooling documentation and packaging clarity.
- Third-party skill-pack attribution and license handling.
- Optional user-installed skill-pack workflows.
- Lightweight CTF solve trace capture.
- Challenge workspace metadata.
- Draft writeup export.

Recommended release wording:

> PawnLogic 0.1.2 focuses on polishing the existing CTF workflow: CTF tooling
> dependency clarity, external skill-pack guidance, upstream attribution,
> solve trace capture, challenge workspace metadata, and writeup export.

## Non-Goals

Do not include these in 0.1.2:

- 0.2.0 versioning.
- Automated penetration-testing platform features.
- Authorized assessment scope models.
- Nuclei, ZAP, OpenVAS, OWASP, NIST, or MITRE mapping work.
- Active web/API/recon scanner adapters.
- Bundling third-party CTF skill packs into PyPI distributions or generated
  release source archives before license review is complete.

## Current Baseline

- Local `main` is synced with `origin/main`.
- Runtime version is `0.1.1`.
- PyPI packaging currently excludes top-level `skills/`.
- `pawnlogic[ctf]` currently installs CTF-related Python/tool dependencies
  only: `pwntools`, `ROPgadget`, and `ropper`.
- Source-checkout `skills/` are local development assets and are not shipped in
  the default PyPI wheel, sdist, or generated release source archive while
  license review is incomplete.
- `~/.pawnlogic/skills` is the runtime location for user-installed skill packs.

## Product Direction

CTF skills should be treated as optional extension packs, not mandatory package
contents.

Correct framing:

> PawnLogic can work with curated CTF skill packs adapted from upstream
> open-source CTF resources, with project-specific integration, packaging, and
> workflow support.

Avoid these claims:

- "PawnLogic self-developed CTF knowledge base"
- "PawnLogic original CTF corpus"
- Any wording that implies upstream content was authored entirely inside this
  repository.

## Packaging Decision

0.1.2 should keep skill packs external by default.

Recommended behavior:

- `pawnlogic`: core agent runtime, provider logic, tools, and skill scanner.
- `pawnlogic[ctf]`: CTF tool dependencies only.
- CTF skill packs: installed explicitly by users through `/sp install <repo>`,
  copied into `~/.pawnlogic/skills`, or used from a source checkout.
- PyPI wheel/sdist/generated release source archive: no third-party CTF skill
  content unless a later task proves license and notice requirements for every
  included file.
- Normal source checkouts may keep local development copies of those skills,
  but unclear-license directories must be marked `export-ignore` before
  publishing a release.

This avoids forcing uncertain third-party content into every installation while
preserving a clear path for users who want CTF skill packs.

## License Gate

Before any third-party skill file is redistributed in a PyPI package, generated
release source archive, Docker image, or generated bundled-skill directory, the
project must record:

- Upstream repository URL.
- Upstream commit or release tag.
- License identifier.
- License file or notice text location.
- Whether redistribution is allowed.
- Which local files are copied or adapted.
- What PawnLogic-specific changes were made.

If a source has no clear redistribution license, it must remain an installation
instruction or user-local import path only.

## Task Chain

### Task 1: Archive 0.1.1 Plan

Move the old `0.1.1` release plan out of active planning and mark it completed.

Acceptance criteria:

- No active document still describes `0.1.1` as unreleased draft work.
- 0.1.2 planning is the only active release plan under `docs/plans/`.

### Task 2: Add Third-Party Notices Framework

Create or update `THIRD_PARTY_NOTICES.md`.

Acceptance criteria:

- The file explains that listed entries are attribution and license-tracking
  records, not permission assumptions.
- Current CTF skill sources are marked as not redistributed in PyPI or
  generated release source archives until license review is complete.
- Any source with unclear license status is explicitly blocked from packaging.

### Task 3: Clarify Documentation

Update README, GUIDE, SECURITY, and matching Chinese docs where applicable.

Acceptance criteria:

- `pawnlogic[ctf]` is documented as dependency tooling only.
- CTF skill packs are documented as optional, user-installed extension packs.
- Documentation does not claim bundled CTF skill content is original to
  PawnLogic.
- English and Chinese README/GUIDE heading structures remain aligned.

### Task 4: Harden Agent Instructions

Update `AGENT.md` so future agents do not reintroduce unsafe packaging claims.

Acceptance criteria:

- Third-party skills require license inventory before redistribution.
- Extras are not described as conditional file packaging.
- Unclear-license skills are install-guidance only.
- Changelog and docs must not say `pawnlogic[ctf]` installs skill markdown.

### Task 5: License Inventory

Audit every tracked CTF skill source.

Candidate local paths:

- `skills/ctf_*`
- `skills/solve_challenge`
- `skills/heap_exploit`
- `skills/demo_stack_overflow`

Acceptance criteria:

- Each path has a source, license, redistribution decision, and notice entry.
- Files with unclear license status are either removed, rewritten from scratch,
  or kept out of all release packages with clear install/import guidance.

### Task 6: CTF Workspace Metadata

Add lightweight challenge metadata stored in the active workspace.

Suggested file:

- `<workspace>/ctf.json`

Suggested fields:

- `challenge_name`
- `category`
- `source`
- `artifacts`
- `remote_targets`
- `flag_candidates`
- `status`
- `created_at`
- `updated_at`

Potential commands:

- `/ctf init`
- `/ctf status`
- `/ctf artifact`
- `/ctf flag`

Acceptance criteria:

- Metadata is local to the workspace.
- No secrets or API keys are recorded.
- Existing sessions and non-CTF workflows are unaffected.

### Task 7: CTF Solve Trace

Extend the existing tool audit flow instead of creating a parallel logger.

Recommended approach:

- Add an optional metadata object to existing audit records.
- Include a `ctf` sub-object only when CTF metadata is active.
- Keep old audit consumers compatible.

Potential CTF fields:

- `challenge_name`
- `category`
- `artifact`
- `flag_candidate`
- `workspace`

Acceptance criteria:

- Existing audit JSONL records remain valid.
- CTF metadata is optional.
- Audit writes still must not block the main flow.

### Task 8: Writeup Draft Export

Generate a Markdown writeup draft from session content, workspace metadata, and
recorded flag candidates.

Suggested command:

- `/ctf writeup`

Suggested output:

- `<workspace>/writeup.md`

Suggested sections:

- Challenge
- Category
- Environment
- Artifacts
- Approach
- Evidence
- Exploit or script
- Flag
- Lessons learned
- Attribution

Acceptance criteria:

- Missing flag candidates produce a draft, not a solved claim.
- Generated Markdown does not include API keys or local absolute secrets.
- The export is deterministic enough for tests.

### Task 9: Tests

Add focused coverage:

- Packaging: wheel/sdist include no third-party CTF skills unless explicitly
  approved by a license inventory test.
- Documentation language policy.
- THIRD_PARTY_NOTICES coverage for any bundled third-party content.
- CTF workspace metadata persistence.
- CTF audit metadata compatibility.
- Writeup export behavior.

### Task 10: Release Preparation

Only after Tasks 1-9:

- Update `CHANGELOG.md`.
- Update README/GUIDE pairs.
- Update `SECURITY.md` if CTF workflow boundaries changed.
- Run the focused checks.
- Bump `config/paths.py:VERSION` from `0.1.1` to `0.1.2`.
- Build and verify artifacts.

## Decisions And Open Questions

Decision: PawnLogic may keep tracked source-checkout `skills/` development
assets, but unclear-license skill-pack directories must be excluded from
generated release source archives with `.gitattributes export-ignore` until
license inventory is complete.

Open questions:

1. Should `/sp install` support a curated allowlist entry for known CTF skill
   repositories?
2. Should the future bundled-skill path, if ever approved, live under
   `pawnlogic/bundled_skills/` instead of top-level `skills/`?
3. Should CTF writeup export be a slash command only, or also a standalone
   library helper for tests and automation?
