# PawnLogic 0.2.3 Autonomous Runtime Reliability And Deepening Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> `superpowers:subagent-driven-development` or
> `superpowers:executing-plans` to implement this plan task-by-task. Track every
> step with the checkboxes in this file. Never mark a remote or release step
> complete without recording its run or release URL.

**Goal:** Ship 0.2.3 as a compatibility-preserving reliability release that
closes confirmed safety gaps, makes custom Providers recover predictably,
turns runtime evaluation into a real bounded test system, restores safe WSL2
Codex automation, and deepens the largest runtime Modules.

**Architecture:** Work in reviewable vertical slices. First lock down release,
path, process, and Provider correctness. Then deepen existing Seams instead of
splitting files by line count: two format Adapters justify Provider transport,
built-in and MCP tools justify Tool Registry metadata, and manual/autosave
callers justify one persistence Interface. Preserve compatibility facades until
all callers and tests move.

**Tech Stack:** Python 3.10+, pytest, Ruff, selective mypy, prompt_toolkit,
http.client/httpx, SQLite, GitHub Actions, JSONL evaluation artifacts,
setuptools, build, twine, and PyPI Trusted Publishing through GitHub OIDC.

## Global Constraints

- Keep `config/paths.py:VERSION` at `0.2.2` until release preparation.
- Do not change public CLI syntax or remove existing commands.
- Preserve DeepSeek's always-active rule and custom Providers' inactive default.
- Preserve model visibility, public stream delta dictionaries, assistant/tool
  message shapes, tool-call ordering, and `reasoning_content` persistence.
- Keep default user mode quiet; detailed diagnostics remain debug-only.
- Do not package third-party Skill Packs in wheels or sdists.
- Do not add telemetry or send evaluation artifacts over the network.
- Real API runs require explicit enablement, a positive call budget, a positive
  wall-clock budget, temporary Runtime Home, and redacted artifacts.
- Docker, browser, and CTF evaluation must remain local and skip cleanly when
  dependencies are absent.
- Use temporary `PAWNLOGIC_HOME` for all tests and smoke runs.
- Do not use broad `# type: ignore`, global mypy strict mode, or behavior changes
  disguised as typing cleanup.
- Do not push, merge, tag, publish, delete remote branches, or create GitHub
  labels unless the maintainer has explicitly authorized that phase.

## Confirmed Risks To Close Before Release

1. The Docs workflow watches obsolete `_CN` filenames instead of `_zh-CN`.
2. Manual workflow dispatch can select production PyPI without a tag/version
   gate equivalent to the tag path.
3. Browser screenshot and MCP binary asset paths have unsafe filename/path
   composition; absolute file writes use the global Workspace instead of the
   active session Workspace.
4. Several host process and Docker operations bypass the shared Operation
   Policy or lack explicit network/destructive authorization.
5. Anthropic-format model fetch uses OpenAI bearer headers, and Provider
   mutation can persist keys before metadata validation.
6. Streaming and non-streaming retry classification differs; retry/timeouts are
   captured at import time; circuit-breaker half-open behavior is incomplete.
7. Runtime eval classifies timeouts after a scenario returns, uses fake
   offline/soak scenarios, hard-codes API call counts, and can overwrite an
   artifact created in the same second.
8. RuntimeContext, RuntimeMetrics, persistence, Tool Registry metadata, and the
   Turn tool loop still have duplicated owners or shallow compatibility state.

## Execution Model

### Branch And PR Rules

- Activate the repository environment once per shell with
  `source .venv/bin/activate`; this WSL2 checkout does not require a bare system
  `python` command.
- Start each PR from the latest green `origin/main`.
- Use one branch per PR and one ownership domain per branch.
- Add characterization tests before moving behavior.
- Keep compatibility wrappers until all internal callers and import tests pass.
- After local targeted checks, open a PR and monitor all checks to completion.
- On failure, inspect failed logs, reproduce locally, patch, push, and continue
  monitoring. Do not merge a red or pending PR.
- Merge dependency chains sequentially. Independent branches may run in
  parallel only when their write sets do not overlap.

### Subagent Waves

- Maximum three implementation subagents concurrently.
- Wave A: PR 2 runs first after PR 1 merges. PR 2 and PR 3 both edit
  `.github/workflows/main_ci.yml` and therefore cannot safely start from the
  same unmerged baseline.
- Wave B: PR 3 and PR 5 may run in parallel after PR 2 merges. PR 4 follows
  PR 3, PR 6 follows PR 5, and PR 7 may run alongside them.
- Wave C: PR 8 follows PR 7; PR 9 follows PR 4 and PR 6; PR 14 may run in
  parallel because it owns only CLI files.
- Wave D: PR 10 follows PR 4 and PR 6; PR 15 follows PR 5 and PR 6; PR 16
  follows PR 4. These branches have disjoint write sets.
- Wave E is intentionally serial because each PR edits `core/session.py`:
  PR 11 follows PR 10, PR 12 follows PR 11, and PR 13 follows PR 12.
- PR 17 and PR 18 are serial integration/release work.
- Every worker must state its file ownership, avoid reverting other workers,
  return changed paths, and provide exact verification output.

### Per-PR Remote Loop

```bash
branch="$(git branch --show-current)"
git push -u origin "$branch"
gh pr create --base main --head "$branch" --fill
pr_number="$(gh pr view "$branch" --json number --jq .number)"
gh pr checks "$pr_number" --watch --interval 10
```

If a check fails:

```bash
gh pr checks "$pr_number"
failed_run="$(gh run list --branch "$branch" --status failure --limit 1 \
  --json databaseId --jq '.[0].databaseId')"
gh run view "$failed_run" --log-failed
```

Patch only the root cause, rerun targeted local checks, push, and watch again.

  bugs/tests gain Locality. Moving lines into pass-through wrappers is not a win.
- Apply the deletion test: if deleting a new Module merely removes complexity,
  it is shallow; if the complexity would reappear in several callers, it earns
  the Seam.
- One Adapter is a hypothetical Seam. Introduce a formal Seam when at least two
  real Adapters exist or when independent testing requires a stable Interface.
- Prefer characterization tests around externally visible behavior before a
  refactor and focused unit tests at the new Interface afterward.
- Do not combine a security behavior correction with unrelated formatting or
  broad typing cleanup.
- A retry count is not a reliability strategy. Classify failures, enforce a
  total deadline, avoid retries after partial output, and expose local evidence.
- A runtime eval scenario is meaningful only if it executes the real path it
  names, enforces its budget, and produces a redacted, reproducible artifact.
- Every PR must leave the repository releasable even when later PRs are delayed.

## Deferred Beyond 0.2.3

- New public CLI commands or a public Provider health UI.
- New Provider integrations.
- A new runtime framework or LangGraph migration.
- Replacing SQLite or introducing a second persistence backend.
- Global strict mypy.
- Automatic production real-API canaries.
- Mutation testing as a required CI gate.
- Splitting `core/memory.py` without a demonstrated second Adapter or smaller
  caller Interface.
