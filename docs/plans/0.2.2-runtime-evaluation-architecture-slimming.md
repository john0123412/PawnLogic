# PawnLogic 0.2.2 Runtime Evaluation And Architecture Slimming Plan

**Goal:** Build a durable runtime evaluation and debugging system while
reducing code bloat in the largest modules through behavior-preserving splits,
shipped through multiple review branches with local targeted tests and green
remote CI before merge.

**Architecture:** Add evaluation infrastructure before adding expensive or
dependency-heavy scenarios. Keep public runtime contracts stable. Split large
modules only by ownership boundary and reason to change, not by arbitrary line
count.

**Tech Stack:** Python 3.10+, pytest with temporary `PAWNLOGIC_HOME`, Ruff,
mypy typed island, GitHub Actions, JSONL evaluation artifacts, setuptools
dynamic versioning, `build`, `twine`, and PyPI Trusted Publishing through
GitHub Actions OIDC.

---

## Release Theme

0.2.2 should make runtime debugging repeatable. The release should provide a
local evaluation harness, bounded optional smoke suites, CI-safe offline checks,
and smaller internal modules that preserve the 0.2.1 public behavior.

## Non-Goals

- Do not change public CLI syntax, commands, or flags.
- Do not change provider/model visibility rules.
- Do not change public provider stream delta dict schema.
- Do not change tool result message shape, assistant message shape, or
  `reasoning_content` persistence rules.
- Do not introduce remote telemetry or send runtime metrics over the network.
- Do not print or persist raw provider keys.
- Do not package third-party skill packs into wheels, sdists, Docker images, or
  generated bundled-skill directories.
- Do not introduce a new runtime framework.
- Do not bump `config/paths.py:VERSION` until release prep.

## Current Baseline

- 0.2.1 is published to PyPI.
- `v0.2.1` exists and points to the 0.2.1 release-prep commit.
- `main` has a 0.2.1 release result record.
- `config/paths.py:VERSION` remains the runtime version source of truth.
- Runtime metrics remain local process state only.
- Existing release guards protect documentation structure, release consistency,
  language policy, dynamic package metadata, and wheel skill-pack exclusion.

## Global Rules

- Read `AGENT.md`, `PROJECT_MEMORY.md`, and relevant ADRs before each broad
  change.
- Keep `PROJECT_MEMORY.md` current when active-plan, release-state,
  architecture, typed-island, or recurring-risk guidance changes.
- Use temporary `PAWNLOGIC_HOME` for pytest and runtime smoke commands.
- Run targeted tests locally before each push.
- After pushing a branch, monitor GitHub Actions until success. If a run fails,
  inspect failed logs, patch, test locally, push again, and repeat.
- Real API smoke is allowed only when explicitly enabled, bounded by call and
  duration limits, and redacted.

## Remote Workflow For Each PR

1. Create a branch.
2. Push the branch.
3. Watch Actions:

   ```bash
   gh run list --branch <branch> --limit 5
   gh run watch <run-id> --exit-status
   ```

4. If failed:

   ```bash
   gh run view <run-id> --log-failed
   ```

5. Patch, test locally, commit, push again.
6. Merge only after green.

## Final Release

After the release branch is green:

```bash
git checkout main
git pull --ff-only origin main
git merge --ff-only <release-branch>
git push origin main
git tag -a v0.2.2 -m "v0.2.2"
git push origin v0.2.2
```

Then monitor the Publish workflow to success and verify PyPI, GitHub Release,
README badge, fresh install, and wheel `skills/` count of 0.
